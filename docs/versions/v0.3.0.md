# OpenKATE v0.3.0：Execution Fabric MVP

> 前置版本：v0.2.0
> 预计工期：3 个 Sprint
> 版本类型：Execution Alpha

## 1. 版本目标

打通第一条真实的 UI + API + State 业务端到端执行链路。

## 2. 用户需求

| ID | 需求 | 验收结果 |
| --- | --- | --- |
| `EXE-001` | QA 可以把 Approved 场景转换为执行计划 | 计划包含步骤、依赖、变量和超时 |
| `EXE-002` | QA 可以运行 Web UI 步骤 | Playwright 实际操作页面并保存截图 |
| `EXE-003` | QA 可以运行 HTTP API 步骤 | 保存脱敏后的 Request、Response 和断言 |
| `EXE-004` | QA 可以运行 PostgreSQL State 检查 | 只读查询并校验最终业务状态 |
| `EXE-005` | 不同通道可以传递变量 | UI 获得的 `orderId` 可供 API/State 使用 |
| `EXE-006` | 用户可以取消、重试和查看实时进度 | 状态变化实时展示 |
| `EXE-007` | 测试账号和数据可以隔离 | 每次运行获得独立租约并自动释放 |

## 3. 服务实现

| 模块 | 实现内容 |
| --- | --- |
| `workflow-service` | `ScenarioExecutionWorkflow`、Activity 重试、取消和补偿 |
| `execution-service` | ExecutionPlan、DAG 校验、Run、Step、变量上下文和资源租约 |
| `executor-ui` | Playwright Python、Browser Context、动作、截图和 Trace |
| `executor-api` | httpx、鉴权、变量模板、JSON/Schema 断言和证据 |
| `executor-state` | PostgreSQL 只读连接、参数化查询和结果断言 |
| `project-service` | 环境连接配置、账号池和危险操作策略 |
| `apps/web` | 执行计划、实时步骤时间线、取消和重试 |

## 4. Workflow

```text
Validate Scenario Version
  -> Acquire Account/Data Lease
  -> Validate Executor Capabilities
  -> Execute DAG Steps
  -> Persist Step Results
  -> Release Lease / Compensation
  -> Complete Run
```

- Worker Task Queue：`executor.ui`、`executor.api`、`executor.state`。
- Workflow 只保存可序列化状态，不直接执行网络或浏览器操作。
- 重试只覆盖幂等 Activity；有副作用步骤必须提供幂等键或补偿。

## 8. 安全约束

- PostgreSQL State Executor 默认只读账号并限制 SQL 类型。
- Secret 只在 Worker 内存中解析，日志和事件只保存引用。
- UI 和 API 外部地址必须通过项目 Allowlist。
- Browser Context、账号和数据集按 `runId` 隔离。
- 所有变量写入前执行敏感字段扫描。

## 9. 测试计划

- DAG 环、缺失变量、超时和取消测试。
- Worker 崩溃后的 Temporal 恢复测试。
- 重复执行命令的幂等测试。
- 凭据不进入日志、事件和 StepResult 的安全测试。
- 同时运行 10 个场景的资源隔离测试。

## 10. 版本验收

- 使用示例电商应用完成 UI 下单。
- API Executor 执行支付回调。
- State Executor 验证订单状态为 `PAID`。
- UI 产生的 `orderId` 正确传递到 API 和 State。
- 用户可以在运行中看到状态、取消任务并重试失败步骤。
- 异常结束后账号、Browser Context 和测试数据租约全部释放。

## 11. 明确不包含

- AI 自动规划、Mobile、GraphQL/gRPC、MQ/Log、Performance 和 Security。
