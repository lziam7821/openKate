# OpenKATE v0.4.0：Evidence & Report

> 前置版本：v0.3.0
> 预计工期：2 个 Sprint
> 版本类型：Closed-loop Alpha

## 1. 版本目标

从“步骤是否执行成功”提升到“是否有足够证据证明业务目标达成”，形成第一条可审计闭环。

## 2. 用户需求

| ID | 需求 | 验收结果 |
| --- | --- | --- |
| `RPT-001` | 用户可以查看业务级最终结论 | 区分 passed、failed、blocked、inconclusive、flaky |
| `RPT-002` | 每个结论可以追溯到证据 | 可查看截图、请求响应、查询结果和 Trace |
| `RPT-003` | 用户可以查看目标/风险/证据覆盖 | 报告显示缺失证据和未覆盖风险 |
| `RPT-004` | 用户可以导出报告 | 支持 HTML、JSON、JUnit XML |
| `RPT-005` | 用户可以比较两次运行 | 展示步骤、证据和耗时差异 |
| `RPT-006` | 用户可以修正失败分类 | 区分产品、环境、数据、执行器和未知失败 |

## 3. 服务实现

| 模块 | 实现内容 |
| --- | --- |
| `report-service` | Evidence Index、Outcome Judgment、Coverage、Report 和 Run Read Model |
| `governance-service` | 确定性失败分类规则、人工修正和审计 |
| `execution-service` | 标准证据上传、Object Storage 引用和 Trace 关联 |
| `apps/web` | 报告详情、证据时间线、覆盖矩阵、运行对比和导出 |

## 4. 结果判定

- Required Evidence 失败时，场景结果为 `failed`。
- Required Evidence 缺失时，结果为 `inconclusive`，不能视为通过。
- 环境不可用且业务断言未执行时，结果为 `blocked` 或 `environment_error`。
- Flaky 由历史指纹和重试结果判定，不能由单次失败直接判定。
- LLM 不参与最终 Passed/Failed 的确定性判定。

## 5. 数据模型

```text
report_schema.reports
report_schema.evidence_indexes
report_schema.coverage_snapshots
report_schema.run_read_models
governance_schema.failures
governance_schema.failure_classifications
```

证据二进制保存在 Object Storage，数据库仅保存元数据、Hash、内容类型、权限和对象引用。

## 8. 前端范围

- Run Summary 和业务结果判定。
- 跨通道执行时间线。
- 截图、Request/Response、查询结果和 Trace 查看器。
- Requirement、Risk、Invariant、Evidence 覆盖矩阵。
- 两次运行差异对比。
- HTML、JSON、JUnit 下载。
- 失败分类修正和修正原因。

## 9. 测试计划

- 每个结果枚举的判定单元测试。
- Evidence Event 重复、乱序和延迟到达测试。
- Object Storage 失败时的降级和重试测试。
- 报告 Token、Cookie、密码脱敏测试。
- JUnit XML Schema 和 CI 解析兼容测试。
- Read Model 全量重建测试。

## 10. 版本验收

- `checkout` 报告关联场景版本、UI 截图、支付请求、DB 结果和 Trace ID。
- 删除一个 Required Evidence 后，结果必须为 `inconclusive`。
- 用户能修正一次错误失败分类，并在审计日志中看到前后值。
- 报告可以导出并被 CI 解析。
- 任意结论可反向追踪到原始证据和执行步骤。

## 11. 明确不包含

- Agent 生成、知识召回、BadCase 规则草稿和 CI 自动触发。
