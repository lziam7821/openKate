# OpenKATE v0.6.0：Knowledge & Governance

> 前置版本：v0.5.0
> 预计工期：3 个 Sprint
> 版本类型：Learning Beta

## 1. 版本目标

建立历史缺陷、历史场景、业务规则和 BadCase 的召回、审批、回放和版本闭环。

## 2. 用户需求

| ID | 需求 | 验收结果 |
| --- | --- | --- |
| `KNO-001` | QA 可以导入历史缺陷和历史用例 | 内容被分类、向量化并保留来源 |
| `KNO-002` | Agent 生成时召回相关知识 | 输出展示召回内容和使用位置 |
| `KNO-003` | 用户可以标记失败为 BadCase | BadCase 绑定运行、证据和人工纠正 |
| `KNO-004` | 系统可以生成规则草稿 | 草稿包含来源、适用范围和预期效果 |
| `KNO-005` | Reviewer 可以审批、发布和回滚规则 | 规则状态和版本可审计 |
| `KNO-006` | 规则发布前可以历史回放 | 展示新增命中、误报和漏报变化 |
| `KNO-007` | 用户可以查看规则效果 | 展示命中率、误报率和最近使用情况 |

## 3. 服务实现

| 模块 | 实现内容 |
| --- | --- |
| `agent-service` | Hybrid Retriever、Reranker、Knowledge Snapshot 和引用注入 |
| `governance-service` | BadCase、Rule、RuleVersion、Approval、Replay Evaluation |
| `report-service` | 失败历史、Flaky 指纹和规则效果 Read Model |
| `asset-service` | 缺陷和历史用例标准化导入 |
| `apps/web` | Knowledge、BadCase、Rule Review、回放和指标页面 |

## 4. 召回流程

```text
Project/Domain Filter
  -> Keyword + Entity Search
  -> pgvector Similarity
  -> Business Object Relation Expansion
  -> Freshness / Severity / Approval Rerank
  -> Deduplicate
  -> Knowledge Snapshot
```

首期 Knowledge Graph 使用 PostgreSQL 关系表，不引入独立图数据库。

## 5. 规则生命周期

```text
draft -> in_review -> approved -> published -> deprecated
                    |             |
                    v             v
                 rejected      rolled_back
```

- 规则不能由 Agent 自动发布。
- 高风险规则需要双人审批。
- 发布前必须在不可变历史运行集上回放。
- 发布后保存命中、误报、漏报和人工纠正指标。

## 9. 测试计划

- 项目和权限过滤，禁止跨项目召回。
- 相同输入和 Knowledge Snapshot 可复现召回结果。
- 规则发布审批和高风险双人审批测试。
- 规则回滚后新任务不再召回旧版本。
- 历史回放不修改原始运行数据。
- 向量索引重建和数据删除传播测试。

## 10. 版本验收

- 导入历史缺陷并在相关场景生成时召回。
- 将一次人工修正的 BadCase 转为规则草稿。
- 规则经过回放、审批和发布后，在下一次生成中命中。
- 回滚规则后，新任务使用前一个有效版本。
- 用户可以解释某条规则来自哪个 BadCase、由谁审批、在哪些任务中命中。

## 11. 明确不包含

- GitHub/GitLab、PR Diff、Mobile 和 Performance/Security 通道。
