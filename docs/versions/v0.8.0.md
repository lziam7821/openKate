# OpenKATE v0.8.0：Extended Channels

> 前置版本：v0.7.0
> 预计工期：4 个 Sprint
> 版本类型：Full-capability Beta

## 1. 版本目标

在不改变核心领域模型的前提下，覆盖 Mobile、扩展 API、State、External 和 Quality 通道。

## 2. 用户需求

| ID | 需求 | 验收结果 |
| --- | --- | --- |
| `CHN-001` | 用户可以运行 Mobile 场景 | Appium 在设备上执行并保存截图、日志 |
| `CHN-002` | 用户可以验证 GraphQL/gRPC | 支持请求、流和契约断言 |
| `CHN-003` | 用户可以验证 MQ/Cache/Log/Trace | 支持异步等待和最终一致性断言 |
| `CHN-004` | 用户可以使用 Mock 和测试数据工厂 | 外部依赖和回调可控、可追溯 |
| `CHN-005` | 用户可以执行 Performance/Security | k6、ZAP 结果进入统一报告 |
| `CHN-006` | 一个场景可以跨五类通道编排 | 变量、证据、超时和补偿保持统一 |

## 3. 服务实现

| 模块 | 实现内容 |
| --- | --- |
| `executor-mobile` | Appium Python Client、设备租约、应用安装、日志和截图 |
| `executor-api` | GraphQL、grpcio、Proto/OpenAPI Contract |
| `executor-state` | Kafka/RabbitMQ/NATS、Redis、Log Provider、Trace Query Adapter |
| `executor-external` | Mock、Callback Inbox、Test Data Provider、第三方沙箱 |
| `executor-quality` | k6/ZAP Process Adapter、Metric/SLO Assertion |
| `execution-service` | Capability Negotiation、异步等待、并行分支和跨通道补偿 |
| `project-service` | Device、Queue、Cache、Log、Metric 连接配置和权限策略 |
| `report-service` | Mobile、State、Performance 和 Security 证据渲染 |

## 4. Executor SDK 约束

所有通道必须实现已有 Python `TestExecutor` Protocol：

```text
capabilities
validate
execute
cancel
```

- 新执行器通过 Capability Registry 注册。
- `execution-service` 不允许添加按通道判断的业务分支。
- Worker 和控制面握手时校验 SDK、Contract 和能力版本。
- Executor Contract Test Kit 是发布强制门禁。

## 5. 通道实现内容

### Mobile

- Android/iOS 设备描述、租约和健康检查。
- App 安装、启动、清理和日志采集。
- Appium Action、截图、Page Source 和网络证据。
- Device Farm Adapter。

### API

- GraphQL Query/Mutation、变量、Error 和 Schema 断言。
- gRPC Unary、Client/Server Streaming 和 Proto Contract。
- 跨协议 Metadata、Trace ID 和变量传递。

### State

- MQ 消息等待、过滤、反序列化和顺序断言。
- Redis Key/TTL/Value 只读检查。
- 日志查询和 Trace Span 断言。
- Polling、Backoff、Deadline 和最终一致性窗口。

### External

- HTTP Mock、Callback Inbox 和请求捕获。
- Test Data Factory、Seed、Snapshot 和 Cleanup。
- 第三方 Sandbox Adapter。

### Quality

- k6 场景、阈值和结果解析。
- OWASP ZAP Baseline/Full Scan Adapter。
- SLO、Metric 和 Error Budget 断言。
- 外部进程 CPU、Memory、Network 和 Deadline 限制。

## 8. 测试计划

- 所有 Worker 运行统一 Executor Contract Test Suite。
- Worker 与控制面版本不兼容测试。
- Device、账号和第三方沙箱异常回收测试。
- MQ 乱序、重复和延迟消息测试。
- k6/ZAP 进程资源和网络隔离测试。
- 新通道不修改 Validation/Report 核心 Domain 的架构检查。

## 9. 版本验收

- 一个场景跨 Mobile、API、MQ、DB、Trace 五个通道执行。
- Performance 和 Security 结果进入同一业务报告。
- 新执行器仅通过 SDK 注册即可被 Execution Fabric 调度。
- 所有租约在成功、失败、取消和 Worker 崩溃后释放。
- 不支持的能力在运行前被拒绝，不进入执行中状态。

## 10. 明确不包含

- 生产级 HA、跨区域容灾、自动扩缩容和 GA 契约冻结。
